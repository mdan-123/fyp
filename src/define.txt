================================================================================
                              1. INTRODUCTION
================================================================================

Scheduler AI is an intelligent scheduling application designed for iOS and 
desktop platforms. It combines Natural Language Understanding (NLU) with 
Large Language Model (LLM) capabilities to provide a conversational interface 
for calendar and task management.

The application allows users to:
- Create, update, delete, and query calendar events via natural language
- Manage tasks with intelligent scheduling and time-blocking
- Set reminders with time-based or location-based triggers (geofencing)
- Sync calendars from Google Calendar and Microsoft Outlook
- Receive AI-powered schedule optimisation recommendations
- Track "Time Debt" - a gamification system for accountability
- Authenticate via Firebase, Passkeys (WebAuthn), or Native Biometrics
- Resolve calendar conflicts when external changes are detected

The core philosophy is an "Escalation Pattern" - every user input first goes 
through fast, local NLU models (ModernBERT). If the local models are confident, 
they execute directly. If they're uncertain, the request silently escalates to 
Gemini 2.0 Flash for more sophisticated processing. The user never sees this 
routing decision - they always receive a coherent response.

TECH STACK:
- Backend: Python 3.12, FastAPI, Firebase Admin SDK
- Frontend: Next.js 14 (App Router), React 18, TypeScript, TailwindCSS
- Mobile: Capacitor 6 (iOS native build via Xcode)
- Database: Firebase Firestore (NoSQL document store)
- AI/ML: ModernBERT (Hugging Face Transformers), Google Gemini 2.0 Flash
- Auth: Firebase Auth, WebAuthn/Passkeys, Native Biometrics (Face ID/Touch ID)
- External APIs: Google Calendar API, Microsoft Graph API, Google Places API

3.2 scheduler_engine.py - NLU Pipeline (~405 lines)
-------------------------------------------------------------------------------

PURPOSE: Core Natural Language Understanding engine using fine-tuned ModernBERT 
models for intent classification and Named Entity Recognition (NER).

CLASS: SchedulerNLU

CONSTRUCTOR PARAMETERS:
- intent_path: Path to fine-tuned intent classification model
- ner_path: Path to fine-tuned NER model
- gemini_api_key: API key for LLM fallback
- confidence_threshold: Minimum sigmoid probability for intent (default 0.85)

INSTANCE VARIABLES:
- tokenizer: AutoTokenizer for intent model
- intent_model: AutoModelForSequenceClassification
- intent_labels: List of 16 intent label strings
- ner_pipe: Token classification pipeline with "simple" aggregation
- gemini_client: Google Gemini API client (optional)
- _time_regex: Compiled regex for time pattern extraction

KEY METHODS:

1. format_entities(raw_ner, text) -> dict
   - Converts raw NER pipeline output into structured entity dictionary
   - Merges adjacent tokens of same entity type using start/end offsets
   - Returns dict with keys: dates, times, people, events, tasks, locations,
     durations, recurrence, reminders, preferences, conditions

2. normalise_time(entities, user_timezone) -> str | None
   - Converts extracted date/time entities to ISO UTC timestamp
   - Handles weekday names ("next Friday", "this Monday")
   - Uses dateparser for complex expressions with RELATIVE_BASE
   - Always returns UTC for database storage
   - Handles timezone-aware conversions

3. _extract_time_entities_regex(text) -> dict
   - Fallback regex extraction when NER fails
   - Time patterns: "3pm", "noon", "midnight", "half past 9"
   - Date patterns: weekdays, "today", "tomorrow", relative dates
   - Duration patterns: "2 hours", "30 mins", "half an hour"

4. calculate_end_time(start_iso, entities, user_timezone) -> str
   - Calculates event end time from start + duration
   - If two times extracted, uses second as end time
   - Default: 1 hour duration
   - Handles cross-midnight events

5. llm_fallback(user_input, reason, user_context, chat_history) -> dict | None
   - Escalates to Gemini 2.0 Flash when local models fail
   - Provides structured prompt with:
     * User's upcoming schedule context
     * Recent chat history for reference resolution
     * Current timestamp and timezone
   - Returns same structure as local NLU for seamless dispatch
   - Includes special rules for bulk operations
   - Handles conversational responses for chat continuations

6. process(text, user_context, chat_history) -> dict
   - MAIN ENTRY POINT for NLU processing
   - Tokenizes input and runs through intent model
   - Applies sigmoid for multi-label classification
   - Runs NER pipeline for entity extraction
   - Validates critical entity requirements per intent
   - Falls back to regex time extraction if NER misses patterns
   - Falls back to LLM if validation fails
   - Returns: {source, intents, confidence_scores, entities, text}

SUPPORTED INTENTS (16 total):
- CREATE_EVENT, UPDATE_EVENT, DELETE_EVENT, QUERY_EVENT
- FIND_FREE_TIME, SUGGEST_TIME, CHANGE_RECURRENCE
- CREATE_TASK, UPDATE_TASK, DELETE_TASK, COMPLETE_TASK, QUERY_TASK
- SET_REMINDER, UPDATE_REMINDER, DELETE_REMINDER
- SET_PREFERENCES

3.3 router.py - Multi-Signal Router (~413 lines)
-------------------------------------------------------------------------------

PURPOSE: Intelligent routing layer that decides whether to handle requests 
locally (ModernBERT) or escalate to LLM (Gemini). Uses multiple cheap signals 
before expensive model inference.

CLASS: MultiSignalRouter

CONSTRUCTOR:
- nlu_engine: SchedulerNLU instance
- centroids_path: Path to intent_centroids.npy file

ARCHITECTURE: Escalation Pattern
- Every input enters NLU pipeline first
- If NLU succeeds (validated, confident) → execute directly
- If NLU fails at any signal → silently escalate to LLM
- Caller never handles routing decisions - only results

COMPLEXITY PATTERNS (with weights):
```python
complexity_patterns = {
    "conditional":        {"weight": 2, "patterns": ["if ", "unless", "depending on", ...]},
    "vague_time":         {"weight": 2, "patterns": ["sometime", "whenever", "eventually", ...]},
    "relative_ops":       {"weight": 2, "patterns": ["move all", "reschedule everything", ...]},
    "conversational":     {"weight": 2, "patterns": ["do you think", "help me plan", ...]},
    "referential":        {"weight": 2, "patterns": ["move it to", "delete it", "those tasks", ...]},
    "cross_domain_action":{"weight": 2, "patterns": ["and update the associated", ...]},
    "multi_step":         {"weight": 1, "patterns": ["and then", "after that", "and also", ...]},
    "bulk_refs":          {"weight": 1, "patterns": ["all my", "every ", "everything", ...]},
}
```

SIGNAL PIPELINE (ordered cheapest to most expensive):

SIGNAL 1: Conversational Continuation Detection (0ms)
- Method: _is_conversational_continuation(text, chat_history)
- Parses last assistant turn from chat history
- If last turn was a scheduling action result → don't apply continuation
- If last turn was conversational AND current input has no scheduling 
  keywords → escalate as chat continuation

SIGNAL 2: General Question Detection (0ms)
- Method: _is_general_question(text)
- Catches out-of-domain queries ("What is the capital of France?")
- Exception: Scheduling queries with question words pass through

SIGNAL 3: Weighted Linguistic Complexity (0ms)
- Method: _get_complexity_score(text)
- Threshold: 2 (any strong signal OR two weak signals)
- Safe phrases excluded: "what is my", "show me my", etc.

SIGNAL 4: Embedding OOD Check (~150ms)
- Method: _embedding_similarity_check(text)
- Compares CLS embedding against stored intent centroids
- Threshold: 0.65 cosine similarity
- Catches inputs unlike any training data

SIGNAL 5: Full NLU Pipeline (~300-500ms)
- Calls SchedulerNLU.process() with chat_history
- Validates intent confidence threshold

SIGNAL 6: Entity Artifact Filter
- Method: _clean_entities(entities)
- Removes NER tokenisation noise (tokens < 3 chars)
- Preserves critical entity types

SIGNAL 7: Per-Intent Validation
- Method: _run_intent_validation(intent, entities, raw_text)
- Validates required slots are filled per intent
- Special validation for SET_REMINDER, DELETE_REMINDER

SIGNAL 8: Confidence Margin Tiebreaker
- For single-intent results only
- Margin threshold: 0.15
- Ensures clear winner vs ambiguous split

KEY METHODS:

1. _is_conversational_continuation(text, chat_history) -> bool
   - Detects chat follow-ups vs fresh scheduling commands
   - Prevents accidental escalation when user continues conversation

2. _is_interrogative_query(text) -> bool
   - Detects questions: "is", "are", "do", "does", ends with "?"

3. _escalate_to_llm(text, reason, stage, user_context, chat_history) -> dict
   - Wraps LLM result with routing metadata
   - Returns consistent dispatchable structure
   - Gracefully handles LLM unavailability

4. evaluate(text, user_context, chat_history) -> dict
   - MAIN ENTRY POINT
   - Runs all signal checks in order
   - Returns dispatchable result with source field:
     - "Local_ModernBERT" = NLU handled it
     - "LLM_Escalation" = LLM handled it
     - "escalation_failed" = Neither available

5. dispatch(nlu_result, user_id, intent_handlers) -> list
   - Executes all validated intents sequentially
   - Each handler receives full entity dict
   - Returns list of {intent, result, status} per intent

-------------------------------------------------------------------------------
3.4 IntentEngine.py - Intent Execution Engine (~2355 lines)
-------------------------------------------------------------------------------

PURPOSE: Database interaction layer. Contains handlers for all 16 supported 
intents, performing actual CRUD operations on Firestore. Includes sophisticated
title resolution and structured exception handling.

CLASS: IntentExecutionEngine

CONSTRUCTOR PARAMETERS:
- db_client: Firestore Client instance
- nlu_engine: SchedulerNLU instance (for embedding-based title matching)
- gemini_api_key: For LLM-based title resolution fallback

TITLE RESOLUTION SYSTEM (5-Tier Cascade):
When user says "reschedule the meeting", the engine must find which event:

1. Alias Resolution - Expands user aliases ("fyp" → "Final Year Project")
2. Substring Match - "meeting" in "Team Meeting" ✓
3. Fuzzy Match (RapidFuzz) - "meating" → "Meeting" (>80% partial_ratio)
4. WordNet Synonyms - "appointment" → finds "Meeting" (via NLTK)
5. Embedding Match - Uses model CLS embeddings for semantic similarity (>0.75)
6. Gemini Fallback - LLM-based resolution when all else fails

CUSTOM EXCEPTIONS (imported from custom_exceptions.py):

AmbiguityError:
- Raised when multiple titles match the query
- Carries: candidates (list of enriched dicts), query, entity_key
- Each candidate includes: id, title, start, end, location, description
- Frontend renders selection UI

SlotConflictError:
- Raised when CREATE_EVENT slot is already booked
- Carries: requested_start/end, suggested_start/end, title
- Frontend offers one-tap booking of suggested slot

KEY HELPER METHODS:

_resolve_with_aliases(title_query, aliases) -> list[str]
- Returns candidate strings including alias expansions
- Handles both forward and reverse alias lookup

_resolve_or_raise(title_matches, title_to_item, query, entity_key)
- 0 matches → returns None (try next resolution stage)
- 1 match → returns full item dict with _doc_id
- 2+ matches → raises AmbiguityError with enriched candidates

_fetch_event_by_id(user_id, doc_id) -> dict | None
- Direct Firestore fetch by document ID
- Used after ambiguity resolution to avoid re-triggering title search

_fetch_task_by_id(user_id, doc_id) -> dict | None
- Same as above but for tasks collection

_is_slot_available(user_id, start_iso, end_iso, exclude_doc_id) -> bool
- Checks if time slot is free in user's calendar
- Uses Firestore range query on "end" field

_find_next_available_slot(user_id, start_iso, end_iso, exclude_doc_id) -> tuple
- Finds next free 30-minute slot starting from given time

PURPOSE: Constraint satisfaction scheduler that finds optimal time slots

CLASSES:

CLASS: TimeSlot

- end: datetime

CLASS: BaseScheduler
Base class with core scheduling logic.

CONSTRUCTOR:

- preferences: List of user preference constraints

KEY METHODS:

_get_day_boundaries(target_date) -> tuple
- Returns (8:00 AM, 10:00 PM) as scheduling window

_is_overlapping(slot_start, slot_end) -> bool
- Checks if slot conflicts with existing events
- Respects travel_time buffer before events

_calculate_gaps(slot_start, slot_end) -> tuple
- Returns (min_gap_before, min_gap_after) in minutes

_fails_hard_constraints(slot_start, slot_end, event_category, gaps) -> bool
- Checks WINDOW constraints (allowed time ranges)
- Checks BUFFER constraints (required gaps)
- Returns True if any hard constraint violated

_score_soft_constraints(slot_start, slot_end, event_category, gaps, 
                        relax_hard_constraints, original_start_dt) -> int
- Starts at 100, deducts points for violations
- Considers: preference windows, buffer preferences, back-to-back penalty
- Preserves original time preference when rescheduling

_search_slots(target_date, duration_minutes, event_category, 
              relax_hard_constraints, original_start_dt) -> TimeSlot | None
- Iterates through day in interval_minutes steps
- Returns highest-scoring non-overlapping slot

find_best_slot(target_date, duration_minutes, event_category, 
               original_start_dt) -> TimeSlot | None
- First tries with hard constraints enforced
- If no slot found, relaxes hard constraints and applies heavy penalty

3.6 task.py - Task Scheduling Engine (~410 lines)
-------------------------------------------------------------------------------

PURPOSE: Intelligent task time-blocking that chunks tasks and schedules them 
across multiple days while respecting constraints.

CLASSES:

CLASS: TaskScheduleRequest (Pydantic)
- user_id: str
- target_date: str
- task_ids: List[str] (empty = schedule all pending)

CLASS: TimeSlot (same as optimiser.py)

CLASS: BaseScheduler (same as optimiser.py)

CLASS: CalendarOptimiser(BaseScheduler)
- inject_routines() - Same as Optimiser.inject_routines()

CLASS: TaskScheduler(BaseScheduler)
- interval_minutes = 5 (finer granularity than BaseScheduler)
- Treats ALL events as blocking (no is_locked check)

METHODS:

_map_task_to_category(task) -> str
- Infers category from task tags and energy_level
- "high" energy → DEEP_WORK
- "MEETING" tag → MEETING
- Default → SHALLOW_WORK

_find_best_slot_across_days(start_date, end_date, duration, category, 
                            due_dt, scheduled_dates) -> TimeSlot | None
- Searches up to 60 days for optimal slot
- Applies spacing penalty to prevent bunching:
  - Same day: -3000 points
  - Next day: -800 points
  - 2 days apart: -300 points

schedule_tasks(start_date, fallback_end_date, pending_tasks) -> List[Dict]
- THE MAIN TASK SCHEDULING ALGORITHM
- Sorts tasks by priority, then due_date
- For each task:
  1. Calculate chunk size based on energy_level
  2. Try scheduling largest chunk that fits
  3. If blocked, halve chunk size and retry
  4. Track scheduled dates for spacing penalty
  5. Generate ghost events linked to task

CHUNK SIZE LOGIC:
- High energy: min 45, max 90 minutes
- Medium energy: min 30, max 120 minutes
- Low energy: min 15, max 60 minutes

3.8 llm_parser.py - Constraint Parser (~120 lines)
-------------------------------------------------------------------------------

PURPOSE: Translates natural language scheduling preferences into structured 
JSON for the constraint satisfaction algorithm.

CLASS: ConstraintParser

FUNCTION: parse(user_text) -> dict | None

OUTPUT STRUCTURE:
{
    "category": "DEEP_WORK | SHALLOW_WORK | MEETING | ...",
    "type": "WINDOW | BUFFER | DURATION",
    "params": { "start": 0-24, "end": 0-24, "minutes": int },
    "is_hard": boolean,
    "is_exclusive": boolean,
    "is_routine": boolean,
    "duration": int | null,
    "weight": 1-10,
    "reasoning": "string"
}

LOGIC RULES:
- "No X after Y" → WINDOW {start: 0, end: Y}
- "No X before Y" → WINDOW {start: Y, end: 24}
- "I need a gap after X" → BUFFER {minutes: 15}
- "Only X during Y" → is_exclusive: true
- "Every day I do X for [duration]" → is_routine: true

WEIGHTING:
- Certain language ("Never", "Must", "Strictly") → is_hard: true, weight: 10
- Suggestive language ("Prefer", "Ideally") → is_hard: false, weight: 1-9

5.3 Centroid-Based OOD Detection
-------------------------------------------------------------------------------

FILE: /src/intent_centroids.npy (256 KB)

PURPOSE: Out-of-distribution detection using embedding similarity.

GENERATION PROCESS:
1. For each intent class, collect 100+ representative training examples
2. Pass each example through ModernBERT encoder
3. Extract [CLS] token embedding (768 dimensions)
4. Compute mean embedding per class → centroid

STORAGE FORMAT:
- NumPy array shape: (16, 768)
- Row i = centroid for INTENT_LABELS[i]

USAGE IN router.py:
```python
# Extract query embedding
query_embedding = self.model.encode(text)  # [768]

# Compute cosine similarity to each centroid
similarities = cosine_similarity(
    query_embedding.reshape(1, -1),
    self.intent_centroids
)[0]

# Best match intent
best_intent_idx = np.argmax(similarities)
best_similarity = similarities[best_intent_idx]

# OOD threshold check
if best_similarity < 0.75:
    return "OOD"  # Out of distribution
```

┌─────────────────────────────────────────────────────────────────────────────┐

│   1. Query all events/tasks with completion_status == "pending"            │

│   4. Aggregate totals into user's debt_ledger document                     │

   - Distinguishes:

     - Refunded: Time recovered through rescheduling
   - Procrastination profiling per user

   - Hard constraints (must be respected)
   - Soft constraints (weighted preferences, can be violated)
   - Exclusive windows ("only deep work from 9-12")
   - Buffer requirements ("15 min after meetings")
   - Category-aware rules

   - Automatic chunking based on energy level

   - Bi-directional Google/Outlook sync
   - Drift detection when external changes occur
   - Three resolution modes:

     2. Keep AI Plan (push to provider)

   - Conflict queue for sequential resolution

G) ROLLBACK CAPABILITY
   - Automatic snapshots before optimisation
   - One-click undo to previous state
